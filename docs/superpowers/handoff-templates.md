# 阶段交接提示词模板

> 本文件为**模板参考**，供填充后打印输出。实际交接词只打印、不落盘、不进 git——新对话恢复上下文唯一依赖是已提交的 spec / plan 文档。

阶段化开发详见 `CLAUDE.md` / `AGENTS.md` 协作规范第 7 条。两个交接点对应两套模板，执行 `brainstorming` / `writing-plans` 收尾时按字段填充后，在回复末尾以代码块输出，然后停下、不得跨阶段继续。

## spec → plan（需求规划 → 实现计划）

```
—— 阶段交接：需求规划 → 实现计划 ——

请新开一个对话，粘贴本提示词作为首条消息继续。

【上阶段产出物】
- spec 文档：<相对路径>
【当前阶段】需求规划已完成
【下一阶段】编写实现计划（使用 writing-plans skill）
【关键上下文】
- 需求要点：<一句话概括目标>
- 已确定的设计取舍：<列出 1-3 条>
- 未决问题：<列出待定项，或写"无">
【下一步动作】
1. 阅读 spec 文档
2. 用 writing-plans skill 产出实现计划，按 TDD + 原子 commit 组织任务
3. 计划文档提交 git，并输出 plan→执行 的交接提示词
```

## plan → 执行（实现计划 → 编码执行）

```
—— 阶段交接：实现计划 → 编码执行 ——

请新开一个对话，粘贴本提示词作为首条消息继续。

【上阶段产出物】
- plan 文档：<相对路径>
【当前阶段】实现计划已完成
【下一阶段】编码执行
【关键上下文】
- 任务总数与分组：<如"16 个任务，分数据层/trait 改造/编排接入/前端/验收 5 组">
- 跨模块/高风险点：<列出 trait 原子改造等需三处同改的点，或写"无">
- 未决问题：<列出待定项，或写"无">
【下一步动作】
1. 阅读 plan 文档
2. 选择执行方式：① 子代理驱动（subagent-driven-development，每任务一个子代理 + 审查）② 内联执行（executing-plans，批量 + 检查点）
3. 若选子代理驱动，先对每个任务执行「强弱模型决策门」：默认弱模型；仅跨模块 trait 改造等架构取舍点判断是否升级强模型复核
```
