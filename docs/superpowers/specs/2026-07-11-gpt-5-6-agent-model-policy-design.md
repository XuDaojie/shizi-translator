# GPT-5.6 子代理模型策略设计

## 目标

将 Codex 子代理从 GPT-5.4 / GPT-5.5 完整迁移到 GPT-5.6 系列，同时保留弱、中、强三档决策框架。Claude Code 与 Grok Build 的现有策略不变。

## 模型映射

| 档位 | Codex 模型 | 默认推理强度 | 定位 |
|---|---|---|---|
| 弱模型 | `gpt-5.6-luna` | `low` | 检索、文档整理、重复检查、脚本化验证、低风险机械修改 |
| 中模型（默认） | `gpt-5.6-terra` | `medium` | 常规实现、普通缺陷修复、普通审查、计划拆解 |
| 强模型 | `gpt-5.6-sol` | `high` | 跨模块契约、安全边界、性能关键路径、复杂调试、架构取舍 |

模型档位由实际模型决定，推理强度是档位内的执行参数，二者不得混用。任务卡必须同时写明模型档位、实际模型和推理强度。

## 调度规则

- 默认使用 `gpt-5.6-terra` + `medium`。
- 低风险、可机械验证的任务可降为 `gpt-5.6-luna` + `low`。
- 高风险或复杂判断任务使用 `gpt-5.6-sol` + `high`，并在任务卡中限定职责。
- `max` 和 `ultra` 不纳入常规三档，只允许主会话显式指定。
- 子代理连续失败 2 次后停止并回报，不得自行更换模型或提高推理强度。
- 模型升级仍通过主会话重新派发任务卡完成，不进行运行时原地切换。

## 文档改动

1. 更新 `docs/agent-model-policy.md` 的 Codex 映射、调度说明和任务卡格式。
2. 更新 `AGENTS.md` 与 `CLAUDE.md` 中的模型调度摘要，并保持两份文件同步。
3. 删除 Codex 的 GPT-5.4 / GPT-5.5 映射和额外“旗舰模型”概念；Claude Code 的 `fable5` 限制保持不变。

## 验收标准

- 三份规则文档不再将 GPT-5.4 或 GPT-5.5 作为 Codex 子代理模型。
- Codex 三档分别映射到 Luna、Terra、Sol，默认档为 Terra。
- 任务卡包含实际模型与推理强度。
- `AGENTS.md` 与 `CLAUDE.md` 除受保护自动维护区域外保持一致。
- 文档中没有占位符、待定项或互相矛盾的升级规则。
