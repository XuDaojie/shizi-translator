# 启动翻译弹窗与服务卡片实时同步设计

## 背景

当前应用启动后默认打开 `main` 窗口的 `settings.html`。翻译弹窗会在加载时调用一次 `get_app_config` 并用启用服务预建翻译卡片，但设置页保存服务启用状态后没有通知弹窗，所以卡片列表只有重启后才更新。

## 目标

1. 软件启动后默认展示翻译弹窗，而不是设置页。
2. 点击翻译弹窗设置按钮时仍能打开设置页。
3. 设置页保存服务启用/关闭后，已打开的翻译弹窗立即同步翻译卡片，无需重启。

## 范围

- 修改 Tauri 窗口配置与窗口打开逻辑。
- 修改配置保存命令，在保存成功后广播配置变更事件。
- 修改翻译弹窗静态 JS，监听配置变更并刷新服务卡片。
- 不迁移翻译页到 Vue，不重做卡片视觉样式，不改翻译执行链路。

## 设计

### 启动窗口

把 `main` 窗口加载页从 `settings.html` 改为 `translate.html`，让应用启动时直接显示翻译弹窗。窗口基础尺寸先沿用当前 `main` 配置，避免引入新窗口生命周期。

### 设置页窗口

`open_settings` 不再复用 `main` 当前页面，而是查找或创建 label 为 `settings` 的窗口，URL 为 `settings.html`。这样启动窗口可以是翻译弹窗，设置按钮仍打开设置页。

### 配置变更事件

`save_app_config` 在 `config_store.save(config)` 成功后，向所有 WebView 广播 `app-config:changed`，payload 使用保存后的 `AppConfig`。该事件只表示“配置已保存”，不负责保存失败、快捷键注册失败等错误状态。

### 翻译卡片刷新

翻译弹窗新增一个复用函数，根据当前配置中的 `services.filter(enabled)` 同步卡片：

- 已启用且不存在的服务：创建占位卡片。
- 已关闭的服务：移除对应卡片。
- 服务顺序变化：按 `services[]` 顺序重排 DOM。
- 服务名称或类型变化：更新卡片标题与图标。
- 当前正在翻译时：不强行清空已有文本；只同步卡片集合，避免打断流式输出。

语言栏继续从配置中的 `defaultSourceLang` 和 `targetLang` 更新。

## 测试

- 前端单测覆盖“启用服务列表变化会新增、删除并排序卡片”的纯函数或可测试辅助函数。
- Rust 构建覆盖新增设置窗口代码。
- 手动验证：启动应用显示翻译弹窗；点击设置按钮打开设置页；开启/关闭服务并保存后，翻译弹窗卡片立即变化。
