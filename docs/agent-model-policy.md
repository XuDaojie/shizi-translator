# 子代理模型调度规范

## 核心规则

调度子代理前必须先确定模型策略。Claude Code 和 Codex 使用弱模型、中模型、强模型三档，默认使用中模型；Grok Build 不参与档位决策，始终继承当前会话模型。

任务卡必须写明实际模型名；Codex 任务卡还必须写明推理强度，禁止只写“弱模型”“中模型”“强模型”。连续失败不是自动升级条件；子代理连续失败 2 次后必须停止并回报主会话。

## 模型映射

| 编码代理 | 弱模型 | 中模型（默认主力） | 强模型 | 旗舰模型 |
|---|---|---|---|---|
| Claude Code | `haiku` | `sonnet` | `opus` | `fable5` |
| Codex | `gpt-5.6-luna` | `gpt-5.6-terra` | `gpt-5.6-sol` | 不适用 |

Codex 默认推理强度随档位固定为：弱模型 `low`、中模型 `medium`、强模型 `high`。模型档位由实际模型决定，推理强度是档位内的执行参数，二者不得混用。`max` 和 `ultra` 不纳入常规三档，只允许主会话显式指定。

## Grok Build 例外

Grok Build 不执行弱模型、中模型、强模型的档位判断，不执行自动升级或降级，始终继承当前会话模型。

Grok Build 任务卡只写：

```md
模型策略：继承当前会话模型
说明：Grok Build 不参与弱/中/强模型决策，不执行自动升级。
```

## 调度规则

- 默认使用中模型：常规实现、普通缺陷修复、普通审查、计划拆解。
- 可使用弱模型：文档整理、检索、重复性检查、脚本化验证、低风险机械修改。
- 使用强模型：跨模块契约、安全边界、性能关键路径、复杂调试、架构取舍。
- `fable5` 只允许主会话显式指定，子代理不得自行升级到旗舰模型。
- Codex 的 `max` / `ultra` 只允许主会话显式指定，子代理不得自行提高推理强度。
- 连续失败 2 次后，子代理必须停止并返回失败摘要、已尝试方案、关键日志，由主会话决定是否换模型、调整推理强度、换代理或重拆任务。

## 升级执行方式

模型升级不是对子代理运行时的原地变更。触发升级条件时，当前子代理必须停止并返回失败摘要、已尝试方案、关键日志和建议后续方向；主会话评估后，决定是否新开一个使用更高模型的新子代理继续处理。

- 若升级目的是复核判断，主会话新开复核子代理，只判断高风险点，不重做全部实现。
- 若升级目的是继续实现，主会话必须重新派发任务卡，带上原任务、失败摘要、关键日志、新的实际模型名；Codex 任务还需写明新的推理强度。
- Grok Build 不参与模型升级流程；触发失败条件时只停机回报主会话，不新开更高档模型子代理。

## 任务卡格式

Claude Code 和 Codex 任务卡必须使用中文字段：

```md
模型档位：中模型
实际模型：gpt-5.6-terra
推理强度：medium
选择理由：常规实现任务，风险可通过测试验证，使用默认主力模型。
```

Claude Code 任务卡不写“推理强度”。如果使用强模型，选择理由必须说明高风险点和强模型的限定职责，例如“仅复核跨模块契约设计”或“仅判断安全边界”。
